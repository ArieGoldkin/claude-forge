Agentic Engineering, and where our stack already lives it

The video's argument, extracted from the full transcript, then diffed against our actual repo — file-cited, not from memory. The unit of work is the AI Developer Workflow (ADW): prompts in → a workflow of code + agents runs → results out. You scale ADWs into a software factory and move your effort to the agentic layer — building the system that builds the system.

So what

We are much closer to this vision than the framing suggests. 12 of 18 concepts are already shipped — often more disciplined than the talk (a benchmarked router, a dedicated refutation agent, enforceable cost governance). The real work is a handful of substrate gaps that stand between "a toolkit of skills" and "a command center that runs ADWs for us."

The recommendation

Close one substrate gap (agent sandboxes) and make one upgrade (router → dispatch full sandboxed ADWs, not just pick a skill). Our router (auto-research) is already benchmarked; wiring it to launch isolated plan→build→test→review pipelines is the shortest path to the command center.

0The thesis: it was never a loop

"Loop engineering is a terrible rebrand of the software development life cycle." [0:20] If we call it loop engineering, we'd need condition-, function-, and exception-engineering too. [8:55]

Loop engineering (the framing he rejects)

Fixate on the loop — an agent + a pass/fail check that routes back. Treats the loop as the whole game. Imprecise, hype-filled, and it hides the real system: it's just one control-flow construct inside the SDLC.

Agentic engineering (the frame he offers)

The unit is the AI Developer Workflow — plan→build→test→review→ship, composed from three actors. The loop is one node. You compose ADWs into a software factory and work the agentic layer, not the app. [1:03, 17:57]

1Three actors, two constraints

"Knowing when and where to place each of these is the name of the game." [3:46] Reliability order: code > engineers > agents. [4:32]

Code#1 reliable

The "unsung hero." Deterministic, zero token cost, no hallucination, runs at light speed. Linters, formatters, type-checkers, tests, hooks. [4:05, 30:28]

Engineers#2

You. Show up at the two constraints — planning and reviewing — and build the system in between. Move up a layer to meta-work. [7:05]

Agents#3

Powerful but least reliable and token-costly. "Don't over-leverage on agents." Pair them with code; specialize them into experts. [30:34]

Constraint A — Planning (the start). The engineer prompts / architects the workflow.
Constraint B — Reviewing (the end). The engineer validates the result. The system handles everything between. [7:18]

2The scaling ladder

One simple workflow, scaled to a software factory. "Add compute to add confidence." [7:47] Rungs tinted teal are where our repo already operates.

L3Collapse validation into a dedicated test agent.7:40
L4Add planning to the front → plan·build·test·review·ship.8:21

L8Factory router agent — reads the goal/codebase, picks the right ADW at best price/perf/speed.13:17
L9Software factory — many specialized ADWs behind a router; runs the app better than any actor alone.17:57

Sequencing rule

Phases 2 and 3 are ordered by a principle from this repo's own history: verification before volume. A silent reviewer froze L3 promotions once already — scaling dispatch (Phase 3) before report recovery and fused validation (Phase 2) would rebuild that failure at factory scale.
