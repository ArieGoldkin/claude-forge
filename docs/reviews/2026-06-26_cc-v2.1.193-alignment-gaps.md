## Tier 1 — Alignment & correctness (do soon)

### T1.1 — Hook `if`-condition file-path fix (v2.1.176) — our hooks depend on it
- **CC change:** v2.1.176 "Fixed hook `if` conditions for file paths with patterns."
- **Our exposure:** `plugins/continuity-toolkit/hooks/hooks.json` uses **6 `if` conditions with file-path glob patterns** — `Write(.env*)`, `Write(*secret*)`, `Write(*.ts) || Edit(*.tsx)`, etc. (security-blocker, auto-approve-project-writes, lint-checker triggers). These gate security and auto-permission behavior.
- **Implication:** these conditions only evaluate correctly on **CC ≥ v2.1.176**. On v2.1.173–175 they could misfire (the bug CC fixed). We benefit automatically on current CC, but our alignment doc understates the floor.
- **Action:** bump the `CC Alignment` baseline in root `CLAUDE.md`; add a note that ctk's file-path `if` conditions require ≥ v2.1.176. Optionally add a smoke check. **No code change needed.**
- **Verified clean:** the separate v2.1.191 bug ("hooks with comma-separated matchers silently never firing") does **not** affect us — all our matchers are single (`Bash`, `Read`) or pipe-alternation (`Write|Edit|MultiEdit`), never comma-separated.

### T1.2 — Model/MCP governance enforcement (v2.1.175, .176, .180, .187) — already-parked follow-up, now fully shipped
- **CC changes:** `enforceAvailableModels` managed setting to constrain the Default model to an allowlist (v2.1.175); `availableModels` enforcement fixes for alias picks (v2.1.176), for subagents and model pickers (v2.1.180), and org-configured model restrictions surfaced in the picker (v2.1.187).
- **Our exposure:** root `CLAUDE.md` "Model economics for subagent dispatch" is **advisory only**. During the auto-research 2.8.0 port (PR #11) I had to *adapt around* the absence of this enforcement layer (the company work-copy already adopted it for v2.1.175–185).
- **Action:** adopt a "Model & MCP governance" subsection into root `CLAUDE.md` — document `enforceAvailableModels` (managed setting, pin allowed model set) + `autoMode.soft_deny: ["Agent(model:fable)"]` as the enforceable backing for the existing advisory guidance. This is the **#1 substantive adoption** and closes the loop on the parked item.

### T1.3 — Auto-mode safety evolution (v2.1.183, v2.1.193) — refresh our auto-mode guidance
- **CC changes:** native **destructive-git-command blocking** in auto mode (v2.1.183); **`autoMode.classifyAllShell`** to route *all* Bash/PowerShell through the classifier (v2.1.193); **denial reasons** now surfaced in transcript, denial toast, and `/permissions` recent-denials (v2.1.193).
- **Our exposure:** root `CLAUDE.md` "Auto Mode Environment" recommends a `hard_deny` baseline (incl. `git push --force*`, `git reset --hard`). CC now blocks destructive git natively, and `classifyAllShell` is a stronger lever than per-pattern denies.
- **Action:** update the Auto Mode section — note that destructive-git blocking is now native (our `hard_deny` git entries are belt-and-suspenders, still valid), add `classifyAllShell` as a recommended hardening lever, and mention denial-reason surfacing for debugging "why was this denied."
