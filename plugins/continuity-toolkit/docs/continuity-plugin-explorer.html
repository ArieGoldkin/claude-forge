Security Guardrails

The protective layer

Prevents Claude from writing to sensitive files or running dangerous commands. Validates every tool call before execution. Cannot be disabled.

Block .env, .ssh/, .aws/credentials, system dirs
Block rm -rf, fork bombs, env dumps
Block Symlink bypass attacks
Audit Every decision logged
Non-disableable Path Resolution Symlink Protection

Lifecycle Orchestration

The glue that connects everything

6 hooks fire at the right moments in the session lifecycle. They read JSON from stdin, do their work, and return JSON to stdout. The infrastructure handles locking, atomic writes, and graceful degradation.

● SessionStart → init + load context
● PreToolUse → security check
● PostToolUse → dirty tracking
● UserPromptSubmit → context warnings
● PreCompact → save to ledger
● SessionEnd → clean shutdown

⚙ Session Lifecycle

Every Claude Code session follows this flow. Hooks fire at each stage, managing state, security, and context tracking automatically.

SessionStart lifecycle/session-loader

Fires when a session begins (startup, resume, clear, or compact).

  • Creates .claude/continuity/ directory structure if missing
  • Checks for stale sessions (was_cleanly_ended === false)
  • Resets dirty tracking (count=0, files=[])
  • Extracts compact ledger summary (~500-800 bytes)
  • Detects current git branch
  • Outputs === SESSION CONTEXT === message to Claude

Tool Usage Loop during session

Repeats for every tool call Claude makes.

  • PreToolUse → pretool/security-blocker validates paths & commands
  • Tool executes (if allowed)
  • PostToolUse → posttool/dirty-state-tracker counts unique file edits
  • UserPromptSubmit → prompt/context-monitor checks context window usage
  • StatusLine updates progress bar, cost, and duration after each response

Context Warnings prompt/context-monitor

Injected invisibly into Claude's context as usage grows.

  • 70% — Advisory: "Consider /create-handoff at stopping point"
  • 80% — Warning: "Recommend handoff then /clear soon"
  • 90%+ — Critical: "Run handoff NOW. Auto-compaction imminent"
  • Rate-limited: only warns when tier increases. Resets after compaction.

PreCompact lifecycle/pre-compact-saver

Fires just before conversation compaction.

  • Appends timestamp marker to the project ledger
  • Format: Auto-saved before compaction: 2026-03-01T12:00:00Z
  • Warns if ledger file not found

SessionEnd lifecycle/session-end

Fires when the session ends (exit, clear, logout).

  • Sets was_cleanly_ended = true in shared-context.json
  • Updates last_activity timestamp
  • Prevents false "stale session" warnings on next start
  • Uses shorter lock timeout (2s vs 5s for other hooks)

💻 Develop — Work Normally

While you work, the plugin hooks run automatically in the background:

Automatic Tracking

  • Security blocker validates every tool call before execution
  • Dirty tracker counts unique files you edit (warns at 15, suggests handoff at 25)
  • Context monitor watches your context window usage and warns Claude at 70/80/90%
  • StatusLine shows a live progress bar with context %, cost, and duration

When to Save

  • After completing 3-5 tasks
  • Before the context window fills up
  • At natural phase transitions (feature done, moving to tests, etc.)
  • Every 1-2 hours during long sessions

💾 /save-state — Update the Ledger

Captures current progress into the project ledger. Uses the append-until-handoff model: some sections are appended, others are replaced as a snapshot.

APPEND — Session Activity Log

  • New timestamped entry added at the top of the log
  • Format: ### YYYY-MM-DD HH:MM with 3-8 bullet points
  • Includes: work done, discoveries, problems/solutions, files modified
  • Never removes previous entries (they accumulate until handoff)

APPEND — Key Decisions

  • New decisions added with format: #### Decision Title (Added YYYY-MM-DD)
  • Each includes: Decision, Reasoning, Impact
  • Never removes existing decisions

REPLACE — Current State (snapshot)

  • Now: Current single focus (branch, status, focus)
  • Done (Recent): Last 10 items (rolling window)
  • Next: Upcoming work priorities
  • Open Questions: Blocking and non-blocking questions

Thresholds

Ledger size: 280 lines
0500 (warn)800 (urgent)1000

📦 /create-handoff — End Session + Clean Ledger

Creates a session handoff document (YAML) and cleans the ledger to keep it lean. This is the critical step that makes the append model sustainable.

Step 1: Create Handoff File

  • Location: .claude/continuity/handoffs/YYYY-MM-DD_topic-name.yaml
  • Format: YAML (~30% fewer tokens than markdown)
  • Contains: accomplished items, pending work, key decisions, files modified, quality checks, blockers

Step 2: Clean the Ledger

  • Session Activity Log: Archived to handoff file, then cleared from ledger
  • Key Decisions >30 days: Moved to archive/ledger-YYYY-QQ.md
  • Done (Recent): Trimmed to last 5 items
  • Current State: Reset to current snapshot

Step 3: Reset Tracking

  • dirty_tracking.files_edited_count = 0
  • dirty_tracking.files_edited_this_session = []
  • continuity.last_handoff updated to new handoff path
  • continuity.last_cleanup updated to current timestamp

▶️ /resume-session — Load Previous Context

Reads the latest handoff, the current ledger, and shared-context.json to synthesize a unified context summary for the new session.

Step 1: Find Latest Handoff

  • Scans .claude/continuity/handoffs/ for .md and .yaml files
  • Sorts by modification time (newest first)
  • If no handoff exists, falls back to ledger + shared-context only

Step 2: Read All Context Sources

  • Handoff: What was accomplished, blockers, recommended next task
  • Ledger: Current State (Now, Done, Next), Key Decisions, Open Questions
  • shared-context.json: Codebase patterns, agent decisions, pending tasks

Step 3: Present Unified Summary

## Session Resumption Summary

### Last Session (from handoff)
- Date: 2026-02-28
- Topic: Auth feature implementation
- Accomplished: Login page, API integration, session mgmt
- Blockers: None

### Current State (from ledger)
- Active Work: Writing tests for auth module
- Key Decisions: JWT with 1hr expiry, httpOnly cookies
- Open Questions: Refresh token strategy

### Recommended Actions
1. Write unit tests for auth middleware
2. Add refresh token endpoint
3. Update API docs

Note: Automatic Context at Session Start

The session-loader hook already loads a compact version of the ledger summary automatically at every session start. /resume-session provides the full context including handoff details — use it when you need the complete picture after a break.

# Project Ledger: my-project /save-state updates timestamp

The header contains the project name, last-updated timestamp, current session description, and the model version (Append-Until-Handoff v2.0).

/save-state: Updates the > Last updated: and > Session: lines.

## Current State → ### Now REPLACED by /save-state

Snapshot of current focus: branch name, what you're working on, and build/test status.

/save-state: Replaces entirely with current snapshot.
/create-handoff: Resets to post-handoff snapshot.
/resume-session: Reads first line as "status" for the compact summary.

### Done (Recent) REPLACED (rolling 10)

Numbered list of recently completed work. Acts as a rolling window.

/save-state: Replaces with last 10 items (rolling window).
/create-handoff: Trims to last 5 items.
/resume-session: Reads last numbered item as "recent" for the compact summary.

### Next REPLACED

Bullet list of upcoming priorities.

/save-state: Replaces with current priorities.
/resume-session: Reads first bullet as "next" for the compact summary.

## Session Activity Log APPENDED by /save-state

Timestamped entries of work done. New entries are added at the top. Never modified or removed by /save-state.

/save-state: Appends new entry at top.
/create-handoff: Archives all entries to handoff YAML, then clears the section.
pre-compact-saver: Appends a compaction timestamp marker.

## Key Decisions APPENDED by /save-state

Architectural and design decisions with reasoning. Never removed by /save-state — only by handoff/archive.

/save-state: Appends new decisions (never removes existing).
/create-handoff: Archives decisions older than 30 days to archive/ledger-YYYY-QQ.md.
/archive-ledger: Same archival logic for decisions >30 days old.

## Open Questions REPLACED

Blocking and non-blocking questions. Replaced as a snapshot each time.

/save-state: Replaces with current questions (blocking / non-blocking).
/resume-session: Displays to help prioritize session work.

## Context Metadata

Branch info, coverage snapshots, and other metadata. Updated with /save-state.

How It Works

  • Reads context_window.used_percentage from stdin JSON
  • Writes percentage to /tmp/claude-context-pct-{sessionId}.txt (read by context-monitor hook)
  • Caches git branch for 5 seconds to avoid repeated shell calls
  • Outputs 2 lines to stdout using ANSI color codes

Locking Protocol

All writers use mkdir-based locking (POSIX atomic) to prevent race conditions.

// Lock path: shared-context.json.lock (directory)
acquireLock(lockPath, maxAttempts)  // mkdir, retry with 100ms delay
  // ... critical section: read → modify → write ...
releaseLock(lockPath)               // rmSync -r

// Timeouts:
//   session-loader:       50 attempts × 100ms = 5s max
//   dirty-state-tracker:  50 attempts × 100ms = 5s max
//   session-end:          20 attempts × 100ms = 2s max

🛡 Security Patterns

The pretool/security-blocker hook blocks writes to sensitive paths and dangerous shell commands. It cannot be disabled via hook-overrides.json.

CategoryPatternBlocked Tool(s)Reason

Symlink Security (ME-001)

The blocker resolves symlinks to prevent bypass attacks:

// Attack: symlink inside project → /etc/passwd
//   project/link.txt → /etc/passwd
//   Write tool targets "project/link.txt" (looks safe)
//   Resolved path = /etc/passwd → BLOCKED

Algorithm:
1. Normalize path (remove ./, collapse //)
2. Check for path traversal (.. remaining = escape attempt)
3. Resolve symlinks via fs.realpathSync
4. Check BOTH normalized AND resolved paths against patterns
5. Either match = DENY

⚙ Provider Compatibility

The plugin is provider-agnostic. It operates at the hooks layer (local filesystem only) and never makes API calls. All hooks work identically regardless of provider.

// The plugin ONLY uses:
//   - stdin  (JSON from Claude Code — same format for all providers)
//   - stdout (JSON back to Claude Code — same format for all providers)
//   - Local filesystem (.claude/, /tmp/, ~/.claude/logs/)
//
// The plugin NEVER:
//   - Makes API calls
//   - Reads API keys
//   - Interacts with the model directly
//   - Branches behavior based on provider (logging only)

function getProviderInfo(): ProviderInfo {
  // Detection is for logging/debugging only
  if (process.env['CLAUDE_CODE_USE_BEDROCK'] === '1')
    return { provider: 'bedrock', ... };
  if (process.env['CLAUDE_CODE_USE_VERTEX'] === '1')
    return { provider: 'vertex', ... };
  if (process.env['CLAUDE_CODE_USE_AZURE'] === '1')
    return { provider: 'foundry', ... };
  return { provider: 'anthropic', ... };
}
