# `mr-{N}-findings.yaml` — canonical schema

The contract between `/etk:review-mr` (writer) and `/etk:post-mr-comments` (reader). Both skills must respect this schema; deviations break the round-trip.

> Path: `.claude/reviews/mr-${MR_NUMBER}-findings.yaml`. Auto-generated by review skill, edited by humans, consumed by post skill.

---

## Top-level fields

```yaml
mr_number: 1612                              # int, required — VCS-native MR/PR number
mr_url: https://gitlab.com/.../merge_requests/1612   # string, required — full URL
project_path: acme/backend/api-service       # string, required — repo namespace path
generated_at: "2026-05-02T14:30:00+03:00"    # ISO8601 with TZ, required
generated_from_sha: 4f3d153eb487f...         # string, required — head_sha at review time
review_mode: deep                            # enum, required — quick | standard | deep
agents_launched:                             # list of strings, required — for traceability
  - "etk:quality-reviewer (code quality)"
  - "dtk:devops-architect (database)"
  - "general-purpose (security)"

# Optional metadata for re-review tracking
previous_review_at: null                     # ISO8601 or null — set if file is being overwritten
previous_findings_count: null                # int or null — useful for diff-vs-previous

findings: []  # list, required — see below
```

---

## `findings[]` entry

Every entry MUST have these fields. Order within the list matters (sort by severity DESC, then confidence DESC).

```yaml
- id: 1                          # int, required — sequential 1-based; stable across edits
  severity: blocking             # enum, required — see Severity enum below
  confidence: 95                 # int 0-100, required — from agent
  path: utils/database/foo.py    # string, required — repo-relative POSIX path, NEW tree
  line: 89                       # int, required — line in NEW (post-diff) file
  body: |                        # string (markdown), required — comment body verbatim
    **blocking:** Phantom table — nothing in this repo writes to `healthy_action_tag`.

    - `grep -rn "HealthyActionTag" --include="*.py"` returns only the model.
    - CMS persists `Action.tags` to DynamoDB.

    _Reviewed with Claude Code._

  # Post-status fields (initialized to false/null by review; mutated by post skill)
  posted: false                  # bool, required
  anchored: true                 # bool, default true — true=inline-anchored; false=posted as top-level note (out-of-hunk fallback) (G9, CC-alignment audit 2026-06-01)
  discussion_id: null            # string or null — populated on successful POST
  note_id: null                  # string or null — populated on successful POST (for delete)
  posted_at: null                # ISO8601 or null — populated on successful POST
```

### Severity enum

| Value | Maps from agent label | UI emoji | Use |
|-------|----------------------|----------|-----|
| `blocking` | `security`, `bug`, `breaking`, `issue [blocking]` | 🔴 | Must fix before merge. Blocks approval. |
| `issue` | `issue [non-blocking]` | 🟡 | Should fix; doesn't block merge. |
| `suggestion` | `suggestion` | 🟢 | Improvement; author judgment whether to apply. |
| `nitpick` | `nitpick` | ⚪ | Cosmetic / preference. |

`praise`-only entries should NOT appear in the YAML; they're rendered in the in-session report only.

### Confidence range

| Range | Tier | When |
|-------|------|------|
| 90-100 | Verified | Tool-confirmed (grep match, lint error, test failure) |
| 70-89 | Pattern match | Documented rule violation with code evidence |
| 50-69 | Suspicious | Looks wrong, no tool verification |
| <50 | Speculative | Should be filtered out before YAML emission |

The review skill drops findings below `$CONFIDENCE_THRESHOLD` (default 70) before writing YAML.

### `body` field conventions

The `body` is written **verbatim** to the MR as the comment body. Required:

- **Conventional-comment label first**: `**blocking:** ...`, `**issue:**`, `**suggestion:**`, `**security:**`, etc. Bold prefix, lowercase, single colon. Allowed: `praise`, `nitpick`, `suggestion`, `issue`, `bug`, `question`, `security`, `breaking`. Pair with `[blocking]` / `[non-blocking]` / `[if-minor]` decorations as needed.
- **2-4 line body** (per `agent-review-templates.md`): one or two sentences of context, then a concrete fix or instruction. Anything longer is a sign the finding should be split or bundled.
- **Mandatory footer**: `_Reviewed with Claude Code._` on its own line at the end. Distinguishes agent-authored comments from human-reviewer comments.

Example (good):

```yaml
body: |
  **blocking:** Audit-trail row is missing for user transitions in `update_state()`.

  `user_audit.py:34` requires an `audit_row` insert on every state change; this branch updates `user.state` without one. Violates I.4.

  _Reviewed with Claude Code._
```

Example (bad — too long, no fix):

```yaml
body: |
  **issue:** This function is hard to follow. There are several places where the logic could be simpler. I noticed that the error handling is also a bit inconsistent. The naming convention used in this file differs from the rest of the codebase. Also, the comments don't really explain why the code is structured this way. Overall I think this could benefit from a refactor when someone has time.
```

The bad example has no concrete action, exceeds the line budget, and bundles unrelated concerns. Reject from agent output rather than emit to YAML.

## Validation rules

`/etk:post-mr-comments` runs these checks before posting and refuses if any fail:

1. **YAML parses** as valid YAML 1.2.
2. **Top-level fields all present**: `mr_number`, `mr_url`, `project_path`, `generated_at`, `generated_from_sha`, `review_mode`, `agents_launched`, `findings`.
3. **Every finding has required fields**: `id`, `severity`, `confidence`, `path`, `line`, `body`, `posted`, `discussion_id`, `note_id`, `posted_at`. `anchored` is **optional**; treat a missing `anchored` as `true` (backward-compatible with pre-G9 YAMLs).
4. **`severity` ∈ enum**: `blocking | issue | suggestion | nitpick`.
5. **`confidence` ∈ [0, 100]** integer.
6. **`line` is positive integer**.
7. **`path` is POSIX-relative** (no `..`, no leading `/`, no Windows separators).
8. **`id` values are unique** within the file.
9. **`body` ends with `_Reviewed with Claude Code._`** (allow trailing whitespace/newline).

Violations cause `STATUS: BLOCKED` from the post skill with a specific error message and the offending finding ID.

## Mutation rules (post skill writes back)

Only `/etk:post-mr-comments` may mutate the YAML, and only the post-status fields:

| Field | Initial | After successful POST |
|-------|---------|----------------------|
| `posted` | `false` | `true` |
| `anchored` | `true` | `true` if posted inline; `false` if posted as a top-level out-of-hunk note (G9) |
| `discussion_id` | `null` | string from API response — GitLab: discussion id; GitHub: the review/issue comment id (which is the thread root) |
| `note_id` | `null` | string from API response — GitLab: `notes[0].id`; GitHub: the same comment id as `discussion_id` |
| `posted_at` | `null` | ISO8601 timestamp at POST time |

> **VCS-agnostic schema.** These two fields are stored as strings for both platforms. GitLab has a
> separate discussion (thread) object and note (comment) object, so the two ids differ. GitHub has no
> distinct thread object — the created comment id IS the thread root — so `discussion_id` and `note_id`
> hold the same value. See `gh-inline-comments-recipe.md` for the GitHub id-mapping table.

**Atomic per-finding**: the post skill writes the YAML after each successful POST, never batch-updates. Lets the file remain accurate if the batch is interrupted (Ctrl-C, network failure, verification failure). Re-running with `--skip` then resumes from the last `posted: true` index.

The post skill MUST NOT mutate any other field. If the user wants to change `body` or `severity`, they edit by hand.

## Re-review handling

When `/review-mr` runs against an MR that already has a `findings.yaml`:

1. Read the existing file, capture its `generated_from_sha` and any `posted: true` entries.
2. Generate the new findings list.
3. **Warn the user** if `posted: true` entries exist whose `path:line` no longer matches a current finding (the comment may now be on a stale anchor; the user may want to mark the discussion resolved).
4. Set `previous_review_at` and `previous_findings_count` from the old file.
5. Overwrite the file with the new findings + new metadata.

**Trade-off note**: this means re-review loses the post-status of any finding that was renumbered. Acceptable: `posted: true` entries are tracked by `discussion_id`, so the user can still query the MR for them. Don't try to do clever finding-matching across review runs — too much room for false matches.

## Markdown render (`mr-{N}-findings.md`)

The post skill does NOT consume the Markdown — it's a human read-only artifact. But the post skill MUST update the per-finding `Posted:` line in the Markdown after each successful POST so the human view stays in sync:

```markdown
### 1. `utils/database/foo.py:89` 🔴 blocking
**Confidence**: 95 | **Posted**: ✅ disc=c6d9fd2d
                                ^^^^^^^^^^^^^^^^
                                Updated by post skill after each successful POST
```

Implementation: after writing to YAML, regenerate the corresponding section of the Markdown by replacing the `Posted: ❌ pending` line. Don't regenerate the entire Markdown (the user may have manually annotated it).
